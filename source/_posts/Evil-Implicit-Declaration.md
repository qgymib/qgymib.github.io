## 前言

隐式声明绝对属于C里面的一个大坑，其特点在于不会引发链接问题，使得在开发项目中如果不注重消除编译告警，则有可能引入意想不到的问题。

## 什么是隐式声明

当一个函数被使用，但是在编译路径上没有找到此函数的原型时，编译器会依据函数调用的参数来生成临时函数原型。这一生成过程被称作隐式声明。

隐式声明的一个特点是其参数随调用者变化，但是其返回值被固定为int。由于这种特点，当一个函数被隐式声明时，很可能引入问题。

## 隐式声明带来的问题

### 参数变化

隐式声明的参数变化是带来问题的一大元凶，且问题常常随着编译平台的变化而变化。

#### 参数位宽变化

此问题在各个平台下都比较容易出现。我们举一个简单的例子：

``` c
// 函数原型
void foo(int64_t n)
{
    printf("%" PRId64 "\n", n);
}
```

当我们通过如下代码来隐式调用此函数时，会发生什么？

``` c
int main(int argc, char* argv[])
{
    foo(1);
    return 0;
}
```

64位平台编译输出：
> 1

32位平台编译输出
> 577753503099781121

此问题曾在系统集成过程中给我们带来过小小的困扰，因为发现参数传递过程中出现了失真。接下来我们可以探究下为什么在32位平台下会有这种异常行为。

首先分析数字`577753503099781121`，将其展开为二进制数字得到
> 1000000001001001011111000000 00000000000000000000000000000001

可以看到实际上低32位里面的值是准确的，但是高32位的值却不是0，这导致了整个值就是异常的。

为什么会有这种现象呢？其本质上是由于程序运行内存结构引起的。在C语言中，函数调用栈由调用者创建与维护。在创建调用栈时，参数需要依照自己的位宽压入调用栈。而在此处，调用者误以为参数位宽为32位，因此将32bit数据压入调用栈中并执行函数调用。而被调用函数从调用栈中取参数时，需要取64bit数据，因此除了调用者压入的32bit数据之外，还额外取了其他数据，导致栈的越界访问。

然而为何64位下就是正常的呢？这实际上是编译器做的优化。在64位环境下，对于数据的访问，若其地址已被对齐（简单来讲，其地址的值能被sizeof(void*)整除），则数据访问效率是最高的。这也是为什么大部分平台下malloc所返回的值是能够被sizeof(void*)整除的原因。

在此用例中，调用者构建调用栈时，为了保持访问效率的最优，会将数据以64bit对齐，因此其参数栈与返回栈之间有32bit的空隙。在测试中这个空隙正好是0，所以值没有问题。

### 返回值截断

返回值截断同参数位宽变化一样，可能引入各种问题。一个典型的例子是函数返回指针的时候：

``` c
void* foo(void)
{
    void* p = malloc(sizeof(int));
    printf("%p\n", p);
    return p;
}
```

在如上代码中返回一块内存区域的地址，为了确保地址范围足够大，此处强制高32位为有效数值。在32位系统下，隐式调用此函数并不会带来什么问题，但是在64位系统下就不一样了。

当处于64环境中，且malloc返回值的高32位非0时，隐式调用将导致地址被截断，调用者仅能获取低32位地址：

``` c
0xffffffff013d0010  // callee 返回地址
0x13d0010           // caller 获取地址
```
